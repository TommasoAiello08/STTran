================================================================================
EXTENDED PROJECT REPORT — STTran / Action Genome / Computer Vision coursework
================================================================================

This document is a natural-language export of what this codebase does, how the
pieces fit together, and what analysis and tooling were built around predicted
scene graphs. It is meant to stand alone for a reader who opens the repository
without the chat history that accompanied its development.


--------------------------------------------------------------------------------
1. SCIENTIFIC AND ENGINEERING CONTEXT
--------------------------------------------------------------------------------

The repository implements research on dynamic scene graph generation on the
Action Genome dataset. The baseline is STTran (Spatial-Temporal Transformer for
scene graphs). The primary research direction documented in README.md is APT
(Anticipatory Pre-Training): a two-stage pipeline (anticipatory pre-training
then fine-tuning) with a Progressive Temporal Encoder and pair tracking across
frames, as in the paper "Dynamic Scene Graph Generation via Anticipatory
Pre-training" (Li et al.), implemented in PyTorch on top of STTran-style
components (Faster R-CNN features, relation heads for attention / spatial /
contact groups, and so on).

In parallel with full training and evaluation, the project invested heavily in
a reproducible path from model predictions to human-readable logs, static
visualizations, JSON exports, and quantitative dataset statistics. That path is
what most of the "analysis" files and the dataset_analysis notebook describe.


--------------------------------------------------------------------------------
2. DATA AND RUNTIME ASSUMPTIONS
--------------------------------------------------------------------------------

Action Genome is expected under a conventional layout (see README.md), with
annotations, frames, and video metadata. The Faster R-CNN checkpoint used by
STTran is not shipped; GloVe embeddings can be auto-downloaded into data/.

The coursework-oriented runs described here focus on batches of videos (not
only "five" in name: limits are controlled by environment variables). The
default mode in run_first5_videos_all_frames.py remains predcls: the model uses
ground-truth boxes and predicts relation predicates with confidence scores.
The same driver also supports sgcls and sgdet (STTRAN_MODE): sgdet runs the
full Faster R-CNN detector plus STTran, which is orders of magnitude slower on
CPU or Apple MPS than predcls. Each frame yields a graph: nodes are entities
with class labels and optional boxes; edges belong to three semantic groups
(attention, spatial, contact) with string predicates and softmax-like scores in
[0, 1].


--------------------------------------------------------------------------------
3. FROM MODEL OUTPUT TO TERMINAL LOGS AND IMAGES
--------------------------------------------------------------------------------

The driver script run_first5_videos_all_frames.py (despite the historical name)
selects videos from the Action Genome split, runs the pretrained STTran graph
model on every annotated frame in each selected video, and materializes two
parallel artefact streams. Environment variables include STTRAN_MODE (predcls,
sgcls, sgdet), STTRAN_CKPT, VIDEO_IDS, FRAME_OFFSET, FRAME_LIMIT, VIDEO_LIMIT,
OUT_VIZ_ROOT, and OUT_LOGS_ROOT so compare runs can write under separate trees
(for example output/logs/compare_mode/...). For sgdet, SGDET_STREAMING defaults
to on: each frame is processed with its own det()+model() pass and the log file
is flushed after every frame so long runs show progress instead of staying at a
header-only log until a full batched forward finishes. SGDET_STREAMING=0 restores
one batched pass over all T frames. The detector honours SGDET_RCNN_CHUNK and
SGDET_VERBOSE for chunked Faster R-CNN progress on the batched path; on MPS, if
SGDET_RCNN_CHUNK is unset, a smaller default chunk is applied to surface progress
sooner. Logging uses pred_labels when the checkpoint exposes them (sgdet). The helper
script run_sgdet_graph_demo.py offers a smaller default sgdet batch with its own
output roots (SGDET_OUT_VIZ, SGDET_OUT_LOGS) so predcls artefact trees are not
overwritten.

First stream — machine-parsable logs:
  STTran/output/logs/<run>/<VIDEO>.log

Each log repeats a block pattern per frame: a node section listing integer ids,
class names, and bounding boxes when available; then a predicted relations
section with attention edges in a dedicated format and spatial/contact lines as
"top-k" lists after pruning with TOPK_SPATIAL and TOPK_CONTACT style limits (the
defaults used when generating logs must stay aligned with any script that
parses them).

Second stream — raster outputs per video directory:
  STTran/output/first5_videos/<VIDEO>/

Typical contents include one PNG per frame showing the graph layout, small
legend text files, a timeline GIF when imageio is available, and optionally an
edge_evolution.png summarizing how predicted edges vary over time for that
video. Layout can be circular (fast) or spring-based; layout can be reused
across frames for speed.

The module viz_terminal_scene_graphs.py is the contract between logs and the
rest of the tooling. It defines typed structures (Node, Edge, FrameGraph),
regular expressions that match the exact print format of the run script, a
parser parse_terminal_log that returns a mapping from frame index to FrameGraph,
and rendering helpers used both by the batch runner and by standalone
re-visualization workflows.


--------------------------------------------------------------------------------
4. JSON EXPORT AND THRESHOLDED RE-RENDERING
--------------------------------------------------------------------------------

export_graphs_json.py reads the same logs and writes graphs.json per video: a
structured dump of nodes and edges per frame for downstream tools or external
notebooks.

rerender_with_thresholds.py demonstrates analysis without touching the GPU model
again: it re-reads logs, applies per-group score thresholds to keep at most one
edge per (source, destination, group), and can regenerate visualizations under
user-controlled stringency (environment variables for thresholds and optional
video subsets).


--------------------------------------------------------------------------------
5. FREQUENCY ANALYSIS, LORENZ CURVES, AND VISUAL STYLE
--------------------------------------------------------------------------------

plot_log_frequencies.py is the quantitative backbone for "how imbalanced is the
data we actually observe in predictions". It distinguishes two normalizations
that answer different questions.

Frame-normalized frequencies count, for each predicate or object class, how many
distinct frames contain at least one instance of that label, divided by total
frames across all logs. That measures prevalence of presence in time.

Assignment-normalized frequencies count every edge or node label occurrence and
divide by total edges (globally or per attention/spatial/contact group) or total
nodes. That measures how mass is split across labels when multiple predictions
per frame exist.

The script generates bar charts (matplotlib Agg) for predicates and objects,
including per-group predicate charts. Bars use thin black edges for separation
instead of heavy white outlines. Output PNG names include predicate_frequency,
object_frequency, edge-normalized variants, and group-specific filenames. The
CLI respects environment variables for log directory, output directory, top-k
for spatial/contact lines in the parser, and optional caps on how many category
bars to draw for readability versus plotting all categories.

A dedicated figure lorenz_label_imbalance.png summarizes inequality across
categories using Lorenz curves on the same underlying counters: categories are
sorted from rarest to most common; the horizontal axis is cumulative fraction of
label types and the vertical axis is cumulative fraction of total count. A
diagonal reference line corresponds to perfect equality; bending far below it
indicates concentration in a few labels. Each subplot reports a discrete Gini
coefficient for quick scalar comparison.

plot_matplotlib_style.py centralizes appearance: built-in matplotlib style
sheets in preference order, plus a sans-serif font stack that degrades gracefully
to DejaVu Sans. Scripts wrap plotting in plot_style_context(); notebooks call
apply_notebook_style() after inline backend setup so figures match between
notebook and batch PNG generation.


--------------------------------------------------------------------------------
6. CONFIDENCE DISTRIBUTIONS AND BOX PLOTS
--------------------------------------------------------------------------------

plot_predicate_score_stats.py walks all logs, collects every edge score grouped
by relation group and predicate string, and produces three boxplot images
(attention, spatial, contact) with a cap on the number of predicates shown for
readability. It also writes predicate_confidence_stats.csv and .json with
empirical mean, standard deviation, quantiles, min, max, and counts per
predicate, including an aggregated "all" group row for cross-checking.

An important refinement is that the y-axis is no longer forced to the full
[0, 1] interval when all scores sit in a narrow band: limits are derived from the
data with padding, still clipped to [0, 1] because scores are probabilistic.
Annotations above boxes were adjusted to respect the dynamic y limits.


--------------------------------------------------------------------------------
7. GRAPH STRUCTURE METRICS FROM LOGS
--------------------------------------------------------------------------------

graph_structure_analysis.py treats each parsed frame as an undirected simple
graph: an edge exists between two nodes if any predicted relation (any group)
links them; scores and predicate names are ignored for this layer. It computes
per-class degree statistics (aggregating node occurrences across frames) and
per-frame spectral summaries: eigenvalue gap of the adjacency matrix and the
second eigenvalue of the normalized Laplacian (algebraic connectivity), with
the standard isolated-vertex convention on the Laplacian diagonal.

The dataset_analysis notebook saves node_degree_by_class.csv and
spectral_per_frame.csv under summary_plots and displays descriptive statistics and
histograms for the spectral columns.


--------------------------------------------------------------------------------
8. THE DATASET_ANALYSIS NOTEBOOK AS A NARRATIVE PIPELINE
--------------------------------------------------------------------------------

dataset_analysis.ipynb is the single entry point for reproducing figures and
tables without re-running inference. It configures repository paths, inserts
STTran on sys.path, applies the shared matplotlib style, and fixes TOPK
parameters to match the logs.

Section 1 calls into plot_log_frequencies: all bar charts plus Lorenz figures,
with inline PNG previews using embedded bytes so Jupyter does not cache stale
previews when files are overwritten at the same path.

Section 1b runs graph_structure_analysis to export CSVs and quick spectral
histograms.

Section 2 invokes plot_predicate_score_stats.py via subprocess for parity with
the command line, then moves CSV, JSON, and boxplot PNGs into summary_plots and
displays the boxplots inline.

Later sections pool scores in memory to print per-group means and standard
deviations, derive example thresholds of the form mean plus a multiple of sigma,
deep-scan Action Genome person_bbox.pkl for multi-person frames (finding only
zero or one person per frame in practice), show per-video edge_evolution
thumbnails, and include Section 6 on star-shaped undirected topologies.

Section 6 replaces a course-provided snippet that referenced an undefined
all_graphs list and mislabelled a "graph isomorphism" check: the logic counts how
often the collapsed undirected prediction graph (one edge per unordered node
pair if any attention, spatial, or contact edge exists, consistent with Section
1b) is a star graph (one hub of degree n-1 and n-1 leaves of degree 1, for n at
least two). The Section 6 code cell is self-contained: it resolves STTRAN_DIR,
adds it to sys.path, imports parse_terminal_log, uses LOGS_DIR from earlier cells
when present or defaults to output/logs/first5_videos, aligns TOPK_S and
TOPK_C with the rest of the notebook or environment defaults, builds log_paths
and all_graphs, then prints counts and the star fraction. To analyse logs from
another run, assign LOGS_DIR to the desired Path in the kernel before executing
Section 6.

The edge evolution grid historically failed when code flattened matplotlib axes
arrays incorrectly (iterating rows of a 2D ndarray produced ndarray objects
instead of Axes). The fix is to build subplots with fig.add_subplot in a loop or
to call a tiny helper plot_edge_evolution_grid.py; the notebook uses the
add_subplot pattern so each axis handle is unambiguously an Axes with imshow
available.

--------------------------------------------------------------------------------
12. HOW TO REPRODUCE THE ANALYSIS STORY END TO END
--------------------------------------------------------------------------------

1) Produce or obtain logs and per-frame PNGs via run_first5_videos_all_frames.py
   with consistent TOPK environment variables.

2) Optionally export graphs.json per video with export_graphs_json.py.

3) Run plot_log_frequencies.py from STTran or execute the notebook Section 1 to
   regenerate all frequency and Lorenz PNGs.

4) Run plot_predicate_score_stats.py or notebook Section 2 for boxplots and CSV
   stats.

5) Execute notebook Section 1b for spectral and degree CSVs, later sections for
   thresholds and GT sanity, the edge evolution grid once edge_evolution.png
   files exist, and Section 6 for the star-topology statistic (or run Section 6
   alone if logs and TOPK settings are already valid).

6) Adjust rerender_with_thresholds.py thresholds if you need stricter visual
   narratives without recomputing neural forward passes.

13. RECENT ENGINEERING SESSION (SGDET, MPS, NOTEBOOK, REPOSITORY)
--------------------------------------------------------------------------------

(Recorded May 2026; consolidates a Cursor-assisted iteration on this codebase.)

The following items were implemented or clarified in a focused development pass
and are reflected in the current tree.

Explicit head selection (AG vs VIDVRD). STTran was extended to support an
additional predicate label space (VIDVRD) while preserving ActionGenome tooling.
To prevent silent mix-ups between label spaces, the STTran forward call now
requires an explicit head argument: head="ag" computes and returns the original
three ActionGenome distributions (attention/spatial/contact), while head="vidvrd"
skips those heads and only produces the shared transformer representation when
return_global_output=True. A small wrapper STTranMultiHead attaches a VIDVRD
predicate classifier to that shared 1936-d representation. All call sites in
scripts that rely on ActionGenome outputs were updated to pass head="ag"
explicitly.

SGDet behaviour and observability. Scene graph detection mode is much heavier
than predicate classification because each step runs the full Faster R-CNN
stack. Earlier runs looked "stuck" because a single batched forward over all
frames could run for many minutes while the log file only contained the header
until the forward completed. Streaming mode addresses this by processing one
frame per det()+model() cycle, appending each frame block to the log and
flushing immediately, with optional console lines per frame. Batched mode remains
available when SGDET_STREAMING is set to 0, false, or no.

Apple MPS robustness. Two fixes avoid common failures on Metal: in the RPN,
tensor reshaping uses reshape instead of view so non-contiguous activations do
not raise a view size error during sgdet inference. In STTran's object classifier
path for sgdet and sgcls, the index tensor built with torch.arange for global
node indices is created on the same device as the box index tensor, fixing a
runtime error when the batch is a single frame on MPS (CPU indices indexing an
MPS tensor).

Operational hygiene. A long-running VS Code Pylint language server process was
observed spinning at high CPU when analysing a very large notebook; terminating
that process is safe and the IDE typically restarts the server. Routine CPU
checks showed no stray training script; heavy usage was more often from the
editor renderer and other desktop applications.

Example command for a short sgdet compare run on one video (from the STTran
directory, after setting AG_DATA_PATH to the Action Genome root): set
STTRAN_MODE=sgdet, VIDEO_IDS to the mp4 name, FRAME_OFFSET and FRAME_LIMIT as
needed, and OUT_LOGS_ROOT / OUT_VIZ_ROOT to separate output trees. Use python -u
for unbuffered stdout. Checkpoint paths default to ckpts/sttran_sgdet.tar when
mode is sgdet unless STTRAN_CKPT overrides.

Repository. Changes were committed and pushed to the remote main branch of the
STTran GitHub repository as part of the same effort, including the notebook
Section 6 work, streaming driver logic, MPS-related fixes, and ancillary scripts
where applicable.

14. CLOSING REMARK
--------------------------------------------------------------------------------

This project combines a modern anticipatory scene graph architecture with a
practical, log-centric evaluation layer: anything expressed in the terminal logs
can be turned into publication-style plots, inequality diagnostics, spectral
fingerprints of predicted topology, and threshold-driven visualizations. The
notebook ties those pieces into a single readable story for coursework or
reports, while scripts remain the source of truth for batch reproducibility.
